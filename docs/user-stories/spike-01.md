# 📌 [01-SPIKE] Asynchronous Processing of `.gdb.zip` Files

> ✍️ **Author:** Paloma Soares  |  🗓️ **Edited:** Apr. 11  |  🔖 **Version:** 1.0  |  ✅ **Status:** `Ready`

---

## 🎯 1. Objective

Technically validate the implementation of an asynchronous ETL pipeline for ingesting `.gdb.zip` files from ArcGIS, ensuring efficient, resilient, and scalable processing through persistence in MongoDB.

---

## ❓ 2. Problem / Technical Uncertainty

Processing geospatial files involves handling large volumes of data (especially the SSDMT layer with approximately 999k records), which may lead to:

* Memory overflow (RAM)
* Celery backend limitations (Redis)
* Cascading pipeline failures
* Accumulation of temporary files on disk
* Difficult orchestration across multiple asynchronous stages

---

### KPI / Success Criteria

* ✅ Download completed without loading the entire file into memory
* ✅ SSDMT layer processed without RAM overflow
* ✅ Full pipeline execution without unhandled failures
* ✅ Correct persistence in MongoDB with geospatial indexes
* ✅ Automatic file cleanup guaranteed in 100% of scenarios
* ✅ Resilient pipeline with retry and rollback mechanisms functioning correctly

## ✅ 4. Spike Completion Criteria

#### Complete ETL Pipeline Execution

| Step           | Action                                                                                                                                                       |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Given that** | a valid `job_id` was generated by the calling system **and** the ArcGIS URL is configured through an environment variable                                    |
| **When**       | the download task is executed                                                                                                                                |
| **Then**       | the file must be downloaded via streaming and saved to `/data/downloads/{job_id}.zip`, validated as a valid ZIP file, and the extraction task must be queued |

| Step           | Action                                                                                                                                                                                                                                  |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Given that** | the ZIP file was successfully downloaded                                                                                                                                                                                                |
| **When**       | the extraction task is executed                                                                                                                                                                                                         |
| **Then**       | the content must be extracted to `/data/tmp/{job_id}/`, the `.gdb` directory dynamically located, and the CTMT, SSDMT, and CONJ layers validated for existence and schema — a `chord` must be triggered with the three processing tasks |

| Step           | Action                                                                                                                                                              |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Given that** | the processing tasks have started                                                                                                                                   |
| **When**       | each layer is processed                                                                                                                                             |
| **Then**       | SSDMT must be processed in batches without loading everything into memory; CONJ must be fully processed with geometry validation; data must be converted to GeoJSON |

| Step           | Action                                                                                                        |
| -------------- | ------------------------------------------------------------------------------------------------------------- |
| **Given that** | all chord tasks completed successfully                                                                        |
| **When**       | the callback is executed                                                                                      |
| **Then**       | the data must be inserted into MongoDB, geospatial indexes created, and the job status updated to `completed` |

| Step           | Action                                                   |
| -------------- | -------------------------------------------------------- |
| **Given that** | the pipeline has finished (successfully or with failure) |
| **When**       | the cleanup task is triggered                            |
| **Then**       | all temporary files must be removed from disk            |

## 🛡️ 5. Rules and Technical Constraints

### 📡 Data Sources (Origin)

* ArcGIS endpoint (via `ARCGIS_GDB_URL` environment variable)
* `.gdb.zip` files containing geospatial data

---

### 🔗 Dependencies

| Technology            | Purpose                                           |
| --------------------- | ------------------------------------------------- |
| **Celery**            | Asynchronous orchestration, retry, chord, signals |
| **Redis**             | Result backend                                    |
| **MongoDB**           | Persistence                                       |
| **Fiona / GeoPandas** | Geospatial data reading                           |
| **Shapely / PyProj**  | Geometric transformations                         |

---

### 🗺️ Data / Specific Rules

**Required layers:**

* `CTMT`
* `SSDMT`
* `CONJ`

**Schema validation per layer** (required columns)

**Processing rules:**

* **SSDMT:**

  * Batch reading (10k)
  * Reprojection to `EPSG:4326`
  * Discard invalid records

* **CONJ:**

  * Geometry validation
  * Conversion to GeoJSON

**Failure rules:**

* HTTP `!= 200` → failure
* Invalid ZIP → failure
* Missing `.gdb` → failure
* Missing layer or invalid schema → failure
* Zero valid records → failure

**Resilience rules:**

* Automatic retry (2 attempts) for network failures
* Mongo rollback in case of persistence errors

**Cleanup rules:**

* Always executed (success or failure)
* Must never raise exceptions
