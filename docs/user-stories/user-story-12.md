## 🎯 1. Overview

* **Objective:** Train a Time Series model (Prophet) using the 5-year historical DEC and FEC data for each electrical grid segment, generate forecasts for the next 3 to 5 years, and plot the results in the PDF report: *No-Sensor Scenario* (current trend without intervention), visually highlighting the grid segments at risk of exceeding ANEEL regulatory limits.
* **Problem:** Currently, the report delivered to the utility engineer only presents historical indicators and the current state — snapshots of the past. Without a future projection based on data, the consultant cannot technically demonstrate when and which grid segments will exceed ANEEL limits, nor quantify the financial impact (fines) that the utility company could avoid by installing the sensors.
* **Success:** The PDF generated by [US-07](/docs/user-stories/user-story-07.md) now includes a new Predictive Analysis section containing time series charts for each critical grid segment, with the ANEEL threshold crossing point clearly highlighted.

## 🎨 2. Visual Behavior

### Visual Behavior

The Predictive Analysis section must be included in the PDF. For each grid segment classified as 🔴 red (Score > 10%), the system generates an individual dual-line chart containing the following mandatory elements:

* **Solid blue line:** Actual DEC historical data (last 5 years)
* **Dashed red line:** Forecast without sensors (current trend projected by Prophet)
* **Transparent red shaded area:** 80% confidence interval of the forecast
* **Dotted gray line:** ANEEL regulatory threshold (DEC target for the grid segment)
* **⚠️ Marker:** Exact point where the no-sensor projection exceeds the ANEEL target

#### Happy Path

1. The ETL pipeline confirms the persistence of historical DEC and FEC data (minimum of 5 years) in MongoDB.
2. The post-ETL trigger from US-07 launches the Predictive Analysis module in the background, in parallel with the other calculations.
3. The system automatically filters grid segments with a Criticality Score > 10% (red level, according to US-01).
4. For each filtered segment, the system retrieves the DEC and FEC historical series from MongoDB.
5. A Prophet model is instantiated and trained individually for each segment.
6. The model generates two outputs: (a) DEC forecast for the next 3 years with an 80% confidence interval and (b) estimated year when the ANEEL threshold will be exceeded.
7. The system applies the compounded reduction formula with sensors based on the latest real measurement.
8. The ROI module calculates the cumulative savings from avoided fines during the forecast period.
9. The plotting module (Matplotlib) generates the charts and converts them into PNG base64.
10. The charts are injected into the HTML/Jinja2 template from US-08 under the new "Predictive Analysis" section.
11. The final PDF includes the complete section without layout issues, cropped text, or axis overlap.

## 🛡️ 4. Business Rules

#### Data Prerequisite

> ⚠️ **Critical Rule:** The model must only be trained if the grid segment has at least **4 years** of DEC and FEC historical data in MongoDB. If a segment has fewer than 4 years, the system must log `WARN: segment X — insufficient data for time series` and omit the predictive chart for that segment from the PDF without breaking the pipeline.

#### Why Prophet?

The system uses the Prophet library (Meta, version ≥ 1.1) for the following technical reasons:

* Performs well with short time series (4–5 annual points), unlike ARIMA and LSTM, which require dozens of observations
* Robust against missing data and outliers (common in ANEEL public datasets)
* Natively generates confidence intervals, strengthening the technical argument in the PDF
* Does not require prior data normalization
* Simple API, fully compatible with the existing Python/FastAPI pipeline
* Explainable model for utility engineers

## ✅ 5. Acceptance Criteria

#### Automatic Training Triggered Post-ETL

| Step           | Action|
| -------------- | ----------------- |
| **Given that** | the ETL confirmed the storage of historical data in MongoDB for a utility company with at least one red-level segment (Score > 10%)  |
| **When**       | the US-07 trigger is activated|
| **Then**       | the system must instantiate and train one Prophet model per red-level segment, in parallel, without blocking the main FastAPI thread |

#### Insufficient Data Handled Without Breaking the Pipeline

| Step           | Action|
| -------------- | ------------------ |
| **Given that** | a red-level segment has fewer than 4 years of historical data in MongoDB|
| **When**       | the predictive module is triggered for that segment|
| **Then**       | the system must log `WARN: segment X — insufficient data (N years, minimum 4)` and omit the chart for that segment from the PDF without generating a 500 error |

#### Mathematically Consistent Forecast

| Step           | Action|
| -------------- | ------------------ |
| **Given that** | the model was trained with DEC historical data showing a growth trend|
| **When**       | the forecast is generated|
| **Then**       | the predicted DEC for the last forecast year must be greater than the first historical year, and the lower confidence interval must never be negative |

#### Chart Correctly Generated in the PDF

| Step           | Action                                               |
| -------------- | ---------------------------------------------------- |
| **Given that** | the models were trained and the scenarios calculated |
| **When**       | the plotting module is executed                      |
| **Then**       | the PNG must be generated and embedded into the PDF  |
