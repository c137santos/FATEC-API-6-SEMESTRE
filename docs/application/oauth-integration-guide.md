# OAuth 2.0 + OIDC Integration Guide

This project functions as an OAuth 2.0 **Authorization Server** with OpenID Connect (OIDC) support. Other internal applications can delegate their users' login to this server â€” the "Login with X" pattern.

---

## Glossary

| Term | Meaning |
|---|---|
| **Authorization Server** | This project. Server responsible for authenticating the user and issuing tokens. |
| **Client / Client App** | The application that wants to use the user's login (e.g., another internal system). |
| **Resource Owner** | The end user who owns the account and approves access. |
| **Authorization Code** | Single-use, short-lived code (60 s) generated after consent. Exchanged for tokens. |
| **Access Token** | Opaque access token that the client app uses to call protected APIs. Expires in 1 hour. |
| **Refresh Token** | Long-lived token used to obtain new access tokens without re-authenticating the user. |
| **ID Token** | JWT signed (HS256) containing the user's identity (`sub`, `email`, `username`). |
| **PKCE** | Proof Key for Code Exchange (RFC 7636). Protection against authorization code interception. Mandatory on this server. |
| **code_verifier** | Random string of 43â€“128 characters generated by the client app before starting the flow. |
| **code_challenge** | SHA-256 of `code_verifier`, encoded in base64url. Sent in the authorization request. |
| **scope** | Set of requested permissions. On this server: `openid`, `email`, `profile`. |
| **sub** | Subject identifier â€” unique and immutable identifier of the user on this server. |
| **OIDC** | OpenID Connect. Identity layer over OAuth 2.0 that adds the ID Token and `/userinfo` endpoint. |
| **Discovery Document** | JSON at `/.well-known/openid-configuration` that describes all server endpoints and capabilities. |
| **redirect_uri** | URL of the client app to which the server redirects after consent. Must be pre-registered. |
| **state** | Random string generated by the client app for CSRF protection. Must be validated in the callback. |

---

## What is this flow?

When a client application wants to authenticate a user, it does **not ask for the password directly**. Instead, it redirects the user to this server, which performs the login and returns a secure code. The application exchanges that code for tokens.

The flow implemented is **Authorization Code + PKCE** (RFC 7636), which is the recommended standard for web and mobile applications. PKCE protects against authorization code interception.

```
User         Client App          This server (Authorization Server)
   |                  |                           |
   |  Clicks "Login"  |                           |
   |----------------> |                           |
   |                  |--- GET /oauth/authorize -->|
   |                  |                           | validates client_id, redirect_uri
   |<----- redirects to login screen -------------|
   |                  |                           |
   | performs login   |                           |
   |------------------------------------------>  |
   |<-------- consent screen -------------------|
   |                  |                           |
   | clicks "Allow"   |                           |
   |------------------------------------------>  |
   |<----- redirects with ?code=XXX ------------|
   |----------------> |                           |
   |                  |--- POST /oauth/token ----> |
   |                  |<-- access_token           |
   |                  |    refresh_token          |
   |                  |    id_token               |
```

## Step-by-Step Integration Guide

### 1. Register your application

Make a call to the registration endpoint to get your application credentials. This only needs to be done once.

**`POST /oauth/clients`**

```http
POST http://localhost:8000/oauth/clients
Content-Type: application/json

{
  "client_name": "My Application",
  "redirect_uris": ["http://my-app.com/callback"],
  "allowed_scopes": ["openid", "email", "profile"]
}
```

**Response:**

```json
{
  "client_id": "a3f9e2b1c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3",
  "client_secret": "guard-this-securely-it-will-not-be-displayed-again"
}
```

> **Important:** the `client_secret` is displayed **only once**. Store it in a secure location.

The available scopes are:

| Scope     | What it unlocks                     |
|-----------|-------------------------------------|
| `openid`  | User's unique identifier (sub)      |
| `email`   | Email address                       |
| `profile` | Username                            |

---

### 2. Generate the PKCE pair

Before starting the flow, generate a `code_verifier` (random string of 43â€“128 characters) and the corresponding `code_challenge` (SHA-256 of the verifier, in base64url).

**Python:**

```python
import secrets
import hashlib
import base64

code_verifier = secrets.token_urlsafe(64)  # 43â€“128 chars

digest = hashlib.sha256(code_verifier.encode()).digest()
code_challenge = base64.urlsafe_b64encode(digest).rstrip(b'=').decode()
```

**JavaScript:**

```js
const array = new Uint8Array(64);
crypto.getRandomValues(array);
const verifier = btoa(String.fromCharCode(...array))
  .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');

const encoder = new TextEncoder();
const data = encoder.encode(verifier);
const digest = await crypto.subtle.digest('SHA-256', data);
const challenge = btoa(String.fromCharCode(...new Uint8Array(digest)))
  .replace(/\+/g, '-').replace(/\//g, '_').replace(/=/g, '');
```

Keep the `code_verifier` â€” it will be used in step 4.

---

### 3. Redirect the user to the authorization screen

Build the URL and redirect the user's browser:

```
GET http://localhost:8000/oauth/authorize
  ?client_id=<your_client_id>
  &redirect_uri=http://my-app.com/callback
  &response_type=code
  &scope=openid email profile
  &state=<random_string_for_csrf>
  &code_challenge=<code_challenge>
  &code_challenge_method=S256
```

**Parameters:**

| Parameter              | Description                                            |
|------------------------|---------------------------------------------------------|
| `client_id`            | The ID received during registration                    |
| `redirect_uri`         | Must be identical to the one registered               |
| `response_type`        | Always `code`                                          |
| `scope`                | Desired scopes, separated by space                     |
| `state`                | Random string; will be returned in callback for CSRF validation |
| `code_challenge`       | Generated in step 2                                    |
| `code_challenge_method`| Always `S256`                                          |

What happens next:
- If the user is **not logged in** â†’ will be redirected to the login screen and returned to the flow automatically.
- If logged in â†’ will see the consent screen with requested permissions.

---

### 4. Receive the code in the callback

After the user approves, the server redirects to the `redirect_uri` with:

```
http://my-app.com/callback?code=XXXXXXXXXXXX&state=<state>
```

Extract the `code` and validate that the `state` matches the one you generated in step 3.

---

### 5. Exchange the code for tokens

**`POST /oauth/token`**

```http
POST http://localhost:8000/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=XXXXXXXXXXXX
&code_verifier=<code_verifier_from_step_2>
&client_id=<your_client_id>
&redirect_uri=http://my-app.com/callback
```

**Response:**

```json
{
  "access_token": "Dd2f2ozLYLZU8DXD...",
  "refresh_token": "7t8P0SvpTgwMObjax...",
  "id_token": "eyJhbGciOiJIUzI1NiJ9...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "openid email profile"
}
```

| Token           | What it's for                                           |
|-----------------|---------------------------------------------------------|
| `access_token`  | Call protected APIs (e.g., `/oauth/userinfo`)          |
| `refresh_token` | Get new `access_token` without re-authenticating user  |
| `id_token`      | JWT with user identity â€” decode to get the claims      |

---

### 6. Read the user's identity

#### Option A â€” Decode the ID Token (no extra call)

The `id_token` is a JWT signed with HS256. Decode the payload (middle part, base64url):

```python
import base64, json

parts = id_token.split('.')
payload = json.loads(base64.urlsafe_b64decode(parts[1] + '=='))
# payload = {"sub": "42", "email": "user@example.com", "username": "alice", ...}
```

> Validate the signature in production using the shared `SECRET_KEY` or via `/oauth/userinfo`.

#### Option B â€” Call `/oauth/userinfo`

```http
GET http://localhost:8000/oauth/userinfo
Authorization: Bearer <access_token>
```

**Response:**

```json
{
  "sub": "42",
  "email": "user@example.com",
  "username": "alice"
}
```

The returned fields depend on the granted scopes:

| Scope     | Returned field |
|-----------|-----------------|
| `openid`  | `sub`           |
| `email`   | `email`         |
| `profile` | `username`      |

---

### 7. Refresh the access token (Refresh)

When the `access_token` expires (after 3600 seconds), use the `refresh_token`:

```http
POST http://localhost:8000/oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=<your_refresh_token>
&client_id=<your_client_id>
```

The response includes a new `access_token`. The previous token is immediately invalidated (token rotation).

## Endpoints Reference

| Method | Endpoint                              | Description                            |
|--------|---------------------------------------|----------------------------------------|
| `POST` | `/oauth/clients`                      | Register new client application        |
| `GET`  | `/oauth/authorize`                    | Start authorization flow               |
| `POST` | `/oauth/authorize`                    | Submit consent (internal)              |
| `POST` | `/oauth/token`                        | Exchange code or refresh for tokens    |
| `GET`  | `/oauth/userinfo`                     | Fetch authenticated user's claims      |
| `GET`  | `/.well-known/openid-configuration`   | Discovery document (endpoints & config)|

---

## Discovery Document

The URL `/.well-known/openid-configuration` returns a JSON with all endpoints and configurations:

```http
GET http://localhost:8000/.well-known/openid-configuration
```

```json
{
  "issuer": "http://localhost:8000",
  "authorization_endpoint": "http://localhost:8000/oauth/authorize",
  "token_endpoint": "http://localhost:8000/oauth/token",
  "userinfo_endpoint": "http://localhost:8000/oauth/userinfo",
  "scopes_supported": ["openid", "email", "profile"],
  "response_types_supported": ["code"],
  "id_token_signing_alg_values_supported": ["HS256"]
}
```

OIDC-compatible libraries can use this URL for auto-configuration.

## Out of scope (not implemented)

- RS256 / JWKS endpoint
- Token revocation (`/oauth/revoke`)
- Client Credentials grant (machine-to-machine)
- Dynamic client registration (RFC 7591)
